# PLAN_0_12_0.md — v0.12.0 Implementation Order

**Milestone:** v0.12.0 — Correctness, Reliability & Developer Tooling
**Status:** ✅ Released 2026-03-28
**Last updated:** 2026-03-28

This document defines the recommended implementation order for all v0.12.0
roadmap items. Sequencing is driven by:

1. **Correctness and safety first** — parser guardrails and quick-win
   correctness fixes before large structural changes.
2. **Dependency order** — developer tooling (DT-1–4) requires no new
   infrastructure and can land early; scalability foundations (A-2, D-4)
   are independent of partitioning enhancements.
3. **Risk front-loading** — EC-01 deep fix and SQLancer fuzzing run in
   early phases so the correctness gate is established before 1.0-prep work
   begins.
4. **Low effort, high value early** — parser guardrails, tiered-scheduling
   default flip, and MERGE template validation ship in Phase 1 as cheap
   quick wins.

## Priority Tiers (Scope Risk)

Total estimated scope is ~35–52+ weeks across all seven phases. If the
milestone runs long, phases are deferrable in the order below. A release
can be cut with only the P0 phases complete.

| Tier | Phases | Contents | Action if time is short |
|------|--------|----------|-------------------------|
| **P0 — Must ship** | 1, 2, 3 | Parser guardrails, diagnostic functions, EC-01 deep fix, TPC-H Q7/Q8/Q9, concurrency stress test | Block release until done |
| **P1 — Target** | 4, 7 | SQLancer fuzzing infra, MERGE dedup profiling | Ship if ready; defer individual items to v0.13.0 if not |
| **P2 — Can slip** | 5, 6, 8 | Columnar tracking (A-2), shared buffers (D-4), partitioning enhancements, dbt macro updates | Defer entire phase to v0.13.0 as a unit; do not ship partial |

> **Slip rule:** Never partially ship Phase 5 or Phase 6 — D-4 depends on
> A-2, and a partial catalog migration is worse than no migration. If Phase
> 5 is deferred, defer Phase 6 with it.

**Goal:** Add four SQL-callable diagnostic functions that surface internal
DVM state in a structured, queryable form. No schema changes to catalog
tables; all functions are pure introspection. Low-risk and high operator
value.

| Item | Description | Effort | Status |
|------|-------------|--------|--------|
| DT-1 | **`explain_query_rewrite(query TEXT)`** — parse a query through the DVM pipeline and return the rewritten SQL plus a list of passes applied (operator rewrites, delta-key injections, TopK detection, group-rescan classification). Useful for debugging unexpected refresh behavior without creating a stream table. | ~1–2d | ✅ |
| DT-2 | **`diagnose_errors(name TEXT)`** — return the last 5 error events for a stream table, classified by type (correctness, performance, config, infrastructure), with a suggested remediation for each class. | ~2–3d | ✅ |
| DT-3 | **`list_auxiliary_columns(name TEXT)`** — list all `__pgt_*` internal columns injected into the stream table's query plan with their purpose (delta tracking, row identity, compaction key). Helps users understand unexpected columns in `SELECT *` output. | ~1d | ✅ |
| DT-4 | **`validate_query(query TEXT)`** — parse and run DVM validation on a query without creating a stream table; return the resolved refresh mode, detected SQL constructs (group-rescan aggregates, non-equijoins, multi-scan subtrees), and any warnings. | ~1–2d | ✅ |

## Phase 5 — Scalability Foundations

**Goal:** Deliver the two highest-impact performance items from the roadmap.
Columnar change tracking eliminates wasted delta processing for wide tables
(50–90% delta-volume reduction). Shared change buffers reduce I/O
multiplication in deployments with many stream tables reading from the same
source.

> ⚠️ **Implement in this order:** A-2 first (no multi-consumer coordination
> needed), then D-4 which builds on A-2's column-subset infrastructure.

| Item | Description | Effort |
|------|-------------|--------|
| A-2 | **Columnar Change Tracking.** Per-column bitmask in CDC triggers; skip rows where no referenced column changed; lightweight UPDATE-only path when only projected columns changed. Expected: 50–90% delta-volume reduction for wide-table UPDATE workloads. Combines with the existing VARBIT `changed_cols` column from v0.11.0 WB-1. | ~3–4 wk |
| D-4 | **Shared Change Buffers.** Single buffer per source shared across all dependent STs; multi-frontier cleanup coordination; static-superset column mode for initial implementation. Requires A-2 to establish the per-column tracking infrastructure. | ~3–4 wk |
| PERF-2 | **Auto-enable `buffer_partitioning` for high-throughput sources.** Add `'auto'` option to `pg_trickle.buffer_partitioning` GUC; heuristic auto-detection: switch to RANGE(lsn) partitioned mode when the change buffer exceeds `compact_threshold` in less than one scheduler tick. | ~1–2 wk |

> ⚠️ D-4 **multi-frontier cleanup is the hardest correctness hazard.** Each
> downstream ST advances its own frontier independently. The shared buffer
> must not be truncated past the minimum frontier across all consumers.
> Use `MIN(consumer_frontier)` in the cleanup query, never `MAX`. Write a
> property-based test with 5+ consumers advancing frontiers in random order
> before merging D-4.

**Phase 5 exit criteria:**
- [ ] A-2: CDC triggers emit per-column bitmask; `scan.rs` skips rows where no referenced column changed; E2E test: wide-table UPDATE hits 50%+ delta-volume reduction vs. baseline
- [ ] D-4: Shared change buffer replaces per-ST buffers for same-source deployments; multi-frontier cleanup never races; property-based test with 5 concurrent consumers passes
- [ ] PERF-2: `buffer_partitioning = 'auto'` activates RANGE(lsn) partitioned mode for high-throughput sources; E2E smoke test validates auto-switch
- [ ] Upgrade SQL `sql/pg_trickle--0.11.0--0.12.0.sql` updated with all new catalog columns (A-2 bitmask, D-4 shared buffer table)
- [ ] `scripts/check_upgrade_completeness.sh` passes after Phase 5 changes are merged
- [ ] `just fmt` clean; `cargo clippy --lib` zero warnings; unit tests pass

## Phase 6 — Partitioning Enhancements

**Goal:** Extend the v0.11.0 RANGE partitioning foundation (A1-1 through A1-4)
to multi-column keys, ALTER support, LIST partitioning, HASH partitioning, and
operational quality-of-life improvements.

> These items are independent of each other and can be parallelised. The
> recommended sequence is A1-1b → A1-1c → A1-1d → A1-3b → PART-WARN.

| Item | Description | Effort |
|------|-------------|--------|
| A1-1b | **Multi-column partition keys.** Extend `partition_by` to accept a comma-separated column list; emit `PARTITION BY RANGE (col_a, col_b)`; extend `extract_partition_range()` to compute min/max tuples; inject composite `BETWEEN (a_min, b_min) AND (a_max, b_max)` predicate into MERGE ON clause. | ~1–2 wk |
| A1-1c | **`alter_stream_table(partition_by => …)` support.** Allow adding or changing the partition key on an existing stream table. Repartition the underlying storage table in-place (`CREATE TABLE … AS SELECT` + rename + drop sequence); update catalog; rebuild MERGE template. | ~1–2 wk |
| A1-1d | **LIST partitioning support.** `PARTITION BY LIST` for low-cardinality columns. Requires a new predicate style (`IN (…)` collecting the distinct partition-key values from the delta) instead of `BETWEEN`. | ~1 wk |
| A1-3b | **HASH partitioning via per-partition MERGE loop.** Predicate injection cannot prune HASH partitions (hash function is not invertible). Implement Approach 2 from PLAN_PARTITIONING_SPIKE.md §3: at refresh time, query which child partitions contain delta rows, then issue one targeted MERGE per affected partition. | ~2–3 wk |
| PART-WARN | **Default-partition growth warning.** Emit a `pgrx::warning!()` when the default catch-all partition of a partitioned stream table contains rows after a refresh, prompting the user to create named partition ranges. | ~1–2d |

> **Auto-partition creation** (TimescaleDB-style automatic chunk management)
> remains a post-1.0 item as stated in PLAN_PARTITIONING_SPIKE.md §10.

**Phase 6 exit criteria:**
- [ ] Multi-column partition key accepted; composite range predicate injected; E2E test passes
- [ ] `alter_stream_table(partition_by => …)` repartitions in-place; data preserved; E2E test passes
- [ ] LIST partitioning: `IN (…)` predicate generated from delta; E2E test passes
- [ ] HASH partitioning: per-partition MERGE loop issues one MERGE per affected partition; E2E test passes
- [ ] Default-partition growth emits `WARNING` with hint to add named partition ranges
- [ ] `scripts/check_upgrade_completeness.sh` passes (Phase 6 introduces no new catalog columns; verify no regressions)
- [ ] `just fmt` clean; `cargo clippy --lib` zero warnings; unit tests pass

## Phase 7 — MERGE Profiling

**Goal:** Gather data-driven evidence for the MERGE deduplication problem.
D-2 (async CDC research spike) is **not in this milestone** — it is moved
to the post-1.0 research backlog due to SPI-in-change-callback infeasibility
and Very High implementation risk (see retraction note below).

| Item | Description | Effort |
|------|-------------|--------|
| G14-MDED | **MERGE deduplication profiling.** Profile how often concurrent-write scenarios produce duplicate key entries requiring pre-MERGE compaction. If ≥10% of refresh cycles need dedup in representative E2E workloads, write an RFC for a two-pass MERGE strategy targeting v0.13.0. | ~3–5d |

> ⚠️ **D-2 retraction consideration.** SPI writes inside logical decoding
> `change` callbacks are **not supported** in PostgreSQL. All row buffering
> must occur in-memory within the plugin's memory context; flush only in the
> `commit` callback. In-memory buffers must handle arbitrarily large
> transactions. The Very High risk rating and SPI-in-change-callback
> infeasibility make the originally proposed design non-functional.
> **Recommendation: move D-2 to the post-1.0 research backlog and do not
> include it in this milestone unless a concrete feasibility study produces
> a written RFC first.**

**Phase 7 exit criteria:**
- [ ] MERGE dedup profiling complete; results documented in `plans/performance/`
- [ ] If dedup rate ≥ 10%: two-pass MERGE strategy RFC written for v0.13.0 consideration
- ~~D2-R~~ ➡️ Deferred to post-1.0 research backlog; no action in v0.12.0

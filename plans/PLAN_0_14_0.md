# PLAN_0_14_0.md — v0.14.0 Implementation Order

**Milestone:** v0.14.0 — Tiered Scheduling, UNLOGGED Buffers & Diagnostics
**Status:** 🚧 In progress
**Last updated:** 2026-04-01

This document defines the recommended implementation order for all v0.14.0
roadmap items.

Sequencing is driven by:

1. **Dependency order** — DIAG-1 signal scoring (DIAG-1a) must precede SPI
   data gathering (DIAG-1b) and SQL surface (DIAG-1c/d); G17-SOAK and G17-MDB
   should run only after C-1 and D-1 are merged so the soak covers the final
   code paths.
2. **Risk front-loading** — Phase 1 ships zero-risk quick-wins (C4, DIAG-2,
   DOC-OPM) in the first few hours so the milestone starts with visible,
   reviewable commits and sets de-risk baselines.
3. **Independent features early** — C-1 (manual tiered scheduling) and D-1
   (UNLOGGED buffers) are structurally independent; either can slip without
   blocking the other.
4. **Stability tests near-last** — G17-SOAK and G17-MDB validate the final
   assembled state; running them before the core features are merged produces
   noise, not signal.
5. **Documentation at end** — G16-PAT and DOC-PDC can be drafted in parallel
   with code phases but should not block the release cut.

## Priority Tiers (Scope Risk)

Total estimated scope is ~5–10 weeks across all seven phases. If the
milestone runs long, phases are deferrable in the order below. A release
can be cut with only P0 phases complete.

| Tier | Phases | Contents | Action if time is short |
|------|--------|----------|-------------------------|
| **P0 — Must ship** | 1, 1b, 2, 3 | Quick polish, error state circuit breaker, manual tier assignment, UNLOGGED buffers | Block release until done |
| **P1 — Target** | 4, 5, 6 | Refresh diagnostics, export API + CLI, stability tests | Ship if ready; defer individual items to v0.15.0 if not |
| **P2 — Can slip** | 7 | Patterns guide + pre-deployment checklist | Defer entire phase to v0.15.0 as a unit |

> **Slip rule:** Do not partially ship DIAG-1. Ship all six sub-items
> (DIAG-1a through DIAG-1f) together or defer the whole group. A function
> that returns signals without the view (DIAG-1d) or without documentation
> (DIAG-1f) is harder to fix later than no function at all.

## Phase 1b — Error State Circuit Breaker

**Goal:** Stop the tight crash loop when a refresh hits a permanent error (e.g.
`function max(jsonb) does not exist`). Today, non-retryable errors reset the
backoff but still retry every scheduler cycle until `max_consecutive_errors`
(default 3) triggers `SUSPENDED` — producing 3 full crash cycles with giant
MERGE SQL logged each time, and the error reason buried in
`pgt_refresh_history` instead of surfaced on the table.

| Item | Description | Effort | Ref |
|------|-------------|--------|-----|
| ERR-1a | **Add `last_error_message TEXT` and `last_error_at TIMESTAMPTZ` columns to `pgt_stream_tables`.** Upgrade migration adds both columns (nullable, no default). | ~1h | — |
| ERR-1b | **On `PermanentFailure`: immediately set `status = 'ERROR'`**, store the error message and timestamp in the new columns, and skip incrementing `consecutive_errors` (the table is already stopped). One permanent failure is enough — it will not self-heal. Retryable errors continue the existing backoff + suspension path unchanged. | ~2–3h | [src/scheduler.rs](../src/scheduler.rs) |
| ERR-1c | **Clear error state on pipeline-regenerating API calls.** `alter_stream_table_impl`, `create_or_replace_stream_table`, and `refresh_stream_table` clear `status` back to `ACTIVE` and null out `last_error_message`/`last_error_at` on success. `resume_stream_table` already accepts `Error` status. | ~1–2h | [src/api.rs](../src/api.rs) |
| ERR-1d | **Expose `last_error_message` and `last_error_at` in the `pgtrickle.stream_tables` view** so users can see the error at a glance via `SELECT * FROM pgtrickle.stream_tables WHERE status = 'ERROR'`. | ~30min | — |
| ERR-1e | **E2E test:** Create a stream table with a query that triggers a permanent error, verify the table enters `ERROR` status after one refresh cycle, verify `last_error_message` is populated, verify `alter_stream_table` with a fixed query clears the error. | ~1–2h | — |

**Phase 1b exit criteria:**
- [x] ERR-1a: `last_error_message` and `last_error_at` columns exist in `pgt_stream_tables`
- [x] ERR-1b: A single permanent refresh failure sets `status = 'ERROR'` immediately; scheduler skips `ERROR` tables; retryable errors are unaffected
- [x] ERR-1c: `alter_stream_table`, `create_or_replace_stream_table`, and `refresh_stream_table` clear error state on success
- [x] ERR-1d: `last_error_message` and `last_error_at` visible in `pgtrickle.stream_tables_info` view (via `st.*`)
- [ ] ERR-1e: E2E test covers permanent-error → ERROR status → fix-via-alter → ACTIVE flow

## Phase 2 — Manual Tiered Scheduling (C-1)

**Goal:** Ship the manual-only subset of tiered refresh scheduling. The
original auto-classification goal (80% scheduler-CPU reduction via
`pg_stat_user_tables`) is retracted because that signal is polluted by
pg_trickle's own internal refresh reads. Scope is reduced to explicit per-ST
tier assignment only; auto-classification via `ExecutorStart/End` hooks is
deferred post-1.0.

> ⚠️ Do **not** use `pg_stat_user_tables` `seq_scan`/`idx_scan` counters for
> any tier signal — pg_trickle's own MERGE scans inflate them, making
> actively-refreshed-but-unread STs appear Warm. See PLAN_NEW_STUFF.md §C-1
> risk analysis.

| Item | Description | Effort | Ref |
|------|-------------|--------|-----|
| C-1a | **`tier` column in `pgt_stream_tables` catalog.** Add `refresh_tier TEXT DEFAULT 'hot' CHECK (refresh_tier IN ('hot','warm','cold','frozen'))`. Upgrade migration: additive `ALTER TABLE … ADD COLUMN IF NOT EXISTS`. | ~2–4h | [plans/performance/PLAN_NEW_STUFF.md §C-1](performance/PLAN_NEW_STUFF.md) |
| C-1b | **`ALTER STREAM TABLE … SET (tier = '…')` SQL surface.** Extend `alter_stream_table()` to accept `tier TEXT DEFAULT NULL` parameter; validate against the allowed set; update catalog. Emit `NOTICE` when demoting a Hot ST to Cold/Frozen in case the user set the interval expecting Hot behaviour. | ~2–4h | [plans/performance/PLAN_NEW_STUFF.md §C-1](performance/PLAN_NEW_STUFF.md) |
| C-1c | **Scheduler tier-aware interval multipliers.** In the background worker tick loop, read `refresh_tier` per ST and apply multipliers: `hot` → configured interval (×1), `warm` → ×2, `cold` → ×10, `frozen` → suspend (skip until manually promoted). Gate the feature with `pg_trickle.tiered_scheduling` GUC (default `off`). | ~2–4d | [plans/performance/PLAN_NEW_STUFF.md §C-1](performance/PLAN_NEW_STUFF.md) |
| C-1d | **E2E tests + upgrade migration.** Tests: Hot ST refreshes on time, Cold ST skips N cycles, Frozen ST never refreshes until promoted, invalid tier name rejected. `sql/pg_trickle--0.13.0--0.14.0.sql`: add `refresh_tier` column + update `alter_stream_table` signature. | ~1d | [plans/performance/PLAN_NEW_STUFF.md §C-1](performance/PLAN_NEW_STUFF.md) |

**Phase 2 exit criteria:**
- [x] `refresh_tier` column exists in `pgt_stream_tables`; defaults to `'hot'`; upgrade migration adds column idempotently
- [x] `ALTER STREAM TABLE my_st SET (tier = 'cold')` succeeds; invalid tier names raise `ERROR`
- [x] Scheduler respects tier multipliers when `pg_trickle.tiered_scheduling = on`; Cold ST skips 9 of every 10 cycles; Frozen ST never refreshes
- [x] `pg_trickle.tiered_scheduling = off` leaves all intervals unchanged (backward-compatible)
- [x] `just fmt` clean; `just lint` zero warnings; `just test-unit` pass
- [ ] `scripts/check_upgrade_completeness.sh` passes
- [ ] E2E tests for tier scheduling

## Phase 3 — UNLOGGED Change Buffers (D-1)

**Goal:** Make change buffer tables optionally `UNLOGGED` to reduce CDC WAL
amplification by ~30%. Default is off (logged, crash-safe). Users who have
measured WAL pressure and accept the crash/standby tradeoff opt in explicitly.

> ⚠️ Read the full D-1 risk analysis in PLAN_NEW_STUFF.md before
> implementing. Key hazards: (1) `ALTER TABLE … SET UNLOGGED` acquires
> `ACCESS EXCLUSIVE` — must be scripted to minimise lock window during
> migration of existing buffers; (2) UNLOGGED tables on standbys are zeroed
> on restart — stream tables on standbys will be stale after any standby
> restart until the next FULL refresh completes.

| Item | Description | Effort | Ref |
|------|-------------|--------|-----|
| D-1a | **`pg_trickle.unlogged_buffers` GUC (default `false`).** `GucContext::Suset`. When `true`, new change buffer tables are created as `CREATE UNLOGGED TABLE`. Existing change buffer tables are not retroactively altered (see D-1c). | ~2–4h | [plans/performance/PLAN_NEW_STUFF.md §D-1](performance/PLAN_NEW_STUFF.md) |
| D-1b | **FULL-refresh trigger on empty buffer after crash.** Ensure the scheduler's post-crash path (empty change buffer + non-zero source rows) correctly resets the ST frontier to zero and enqueues a FULL refresh. Add targeted test: truncate change buffer table manually; assert FULL refresh fires on next tick. | ~1d | [plans/performance/PLAN_NEW_STUFF.md §D-1](performance/PLAN_NEW_STUFF.md) |
| D-1c | **Upgrade migration: `ALTER TABLE … SET UNLOGGED` for existing buffers.** In `sql/pg_trickle--0.13.0--0.14.0.sql`, do **not** auto-alter existing buffer tables (too broad, unknown lock impact). Instead, provide `pgtrickle.convert_buffers_to_unlogged()` as an optional operator-run function that iterates `pgtrickle_changes.changes_*` tables and issues `ALTER TABLE … SET UNLOGGED` per table. Document the lock window and advise running during low-traffic windows. | ~1d | [plans/performance/PLAN_NEW_STUFF.md §D-1](performance/PLAN_NEW_STUFF.md) |
| D-1d | **E2E tests.** New buffer created UNLOGGED when GUC is on; buffer created logged when GUC is off; crash-recovery FULL-refresh path triggered; `convert_buffers_to_unlogged()` converts existing logged buffers. | ~1d | [plans/performance/PLAN_NEW_STUFF.md §D-1](performance/PLAN_NEW_STUFF.md) |
| D-1e | **Documentation: staleness window disclaimer.** Add an explicit caveat to `docs/CONFIGURATION.md §unlogged_buffers` describing the temporary staleness window after crashes and standby restarts. | ~1–2h | [plans/performance/PLAN_NEW_STUFF.md §D-1](performance/PLAN_NEW_STUFF.md) |

**Phase 3 exit criteria:**
- [x] `pg_trickle.unlogged_buffers = true` causes newly created change buffer tables to be `UNLOGGED` (verified via `pg_class.relpersistence = 'u'`)
- [x] `pg_trickle.unlogged_buffers = false` (default) leaves change buffer tables logged — backward-compatible
- [x] Empty change buffer after crash detected via postmaster restart comparison → marks ST for reinit (FULL refresh)
- [x] `pgtrickle.convert_buffers_to_unlogged()` function converts all existing logged change buffers; documented with lock-window warning
- [x] `just fmt` clean; `just lint` zero warnings; `just test-unit` pass
- [ ] `scripts/check_upgrade_completeness.sh` passes
- [ ] E2E tests for UNLOGGED buffers and crash recovery

## Phase 4 — Refresh Mode Diagnostics (DIAG-1)

**Goal:** Give operators a structured SQL API to understand whether their
stream tables are running in the most efficient refresh mode. The advisory
function synthesises 7 weighted signals into a composite recommendation with
a confidence level and a human-readable explanation.

> Implement in sub-task order: pure logic first (DIAG-1a), then SPI data
> gathering (DIAG-1b), then SQL surface (DIAG-1c/d), then tests (DIAG-1e),
> then docs (DIAG-1f). Do not ship DIAG-1c before DIAG-1b is passing unit
> tests.

| Item | Description | Effort | Ref |
|------|-------------|--------|-----|
| DIAG-1a | **`src/diagnostics.rs` — pure signal-scoring functions.** `score_change_ratio`, `score_empirical_timing`, `score_query_complexity`, `score_target_size`, `score_index_coverage`, `score_latency_variance`, `compute_recommendation`. All pure Rust, no SPI — fully unit-testable without a DB. Signal weights: historical change ratio 0.30, empirical timing 0.35, current change ratio 0.25, query complexity 0.10, target size 0.10, index coverage 0.05, P95/P50 variance 0.05. Confidence degrades gracefully when `pgt_refresh_history` rows are sparse (< 10 cycles). | 1–2d | [PLAN_DIAGNOSTICS_FUNCTION.md §6 D-1](performance/PLAN_DIAGNOSTICS_FUNCTION.md) |
| DIAG-1b | **SPI data-gathering layer.** Fetch from: change buffer row counts, `pgt_refresh_history` FULL/DIFF timing aggregates, query complexity score from DVM parser (`estimate_query_complexity()`), `pg_relation_size()` for target table, `pg_index` coverage (fraction of GROUP BY / JOIN columns covered by an index). Package into `DiagnosticsInput` struct consumed by DIAG-1a functions. | 1–2d | [PLAN_DIAGNOSTICS_FUNCTION.md §6 D-2](performance/PLAN_DIAGNOSTICS_FUNCTION.md) |
| DIAG-1c | **`pgtrickle.recommend_refresh_mode(name TEXT)` and `pgtrickle.recommend_refresh_mode()` (set-returning) SQL functions.** Read-only, no side effects. Returns columns: `stream_table_name TEXT`, `recommended_mode TEXT`, `confidence TEXT` (`'high'`/`'medium'`/`'low'`), `reason TEXT`, `signals JSONB`. | 0.5–1d | [PLAN_DIAGNOSTICS_FUNCTION.md §6 D-3](performance/PLAN_DIAGNOSTICS_FUNCTION.md) |
| DIAG-1d | **`pgtrickle.refresh_efficiency` view.** Per-table raw data: FULL vs DIFF median timing, change ratios, speedup factor, index coverage fraction. Programmatic companion to the advisory function — suitable for monitoring dashboards and alerting. | 0.5–1d | [PLAN_DIAGNOSTICS_FUNCTION.md §6 D-6](performance/PLAN_DIAGNOSTICS_FUNCTION.md) |
| DIAG-1e | **E2E integration tests + upgrade migration.** Tests: recommendation returned for simple, complex, and sparse-history STs; `signals` JSONB contains all 7 keys; `refresh_efficiency` view returns correct columns; `recommend_refresh_mode()` (no-arg) returns one row per ST. Upgrade script: additive — no catalog DDL changes required. | 0.5–1d | [PLAN_DIAGNOSTICS_FUNCTION.md §6 D-5](performance/PLAN_DIAGNOSTICS_FUNCTION.md) |
| DIAG-1f | **Documentation.** `docs/SQL_REFERENCE.md`: `recommend_refresh_mode` + `refresh_efficiency` reference entries. `docs/CONFIGURATION.md`: `agg_diff_cardinality_threshold` tuning note. `docs/tutorials/tuning-refresh-mode.md`: worked example from observation to ALTER. | ~0.5d | [PLAN_DIAGNOSTICS_FUNCTION.md §6 D-7](performance/PLAN_DIAGNOSTICS_FUNCTION.md) |

**Phase 4 exit criteria:**
- [ ] All 7 signal functions implemented in `src/diagnostics.rs`; unit tests cover normal, sparse-history, and edge-case inputs
- [ ] SPI layer gathers all required data without panicking on missing `pgt_refresh_history` rows
- [ ] `pgtrickle.recommend_refresh_mode('my_st')` returns a single row with `recommended_mode`, `confidence`, `reason`, `signals` JSONB with all 7 keys
- [ ] `pgtrickle.recommend_refresh_mode()` (no-arg) returns one row per registered stream table
- [ ] `pgtrickle.refresh_efficiency` view published and queryable
- [ ] `scripts/check_upgrade_completeness.sh` passes (new functions + view additive)
- [ ] `docs/tutorials/tuning-refresh-mode.md` published

Assert: worker quotas are per-database (one database's refresh load cannot starve the other); shared-memory state (ring buffer, DAG cache) does not bleed between databases. | ~1–2d | [plans/performance/REPORT_OVERALL_STATUS.md §17](performance/REPORT_OVERALL_STATUS.md) |

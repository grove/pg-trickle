This document defines the recommended implementation order for all v0.11.0
roadmap items. Sequencing is driven by:

1. **Correctness and safety first** — panics and silent failures must be
   eliminated before adding new code paths.
2. **Dependency order** — later phases build on infrastructure introduced by
   earlier ones (e.g. WB-1 must land before ST-to-ST; WAKE-1 before C3-1).
3. **Risk front-loading** — design spikes for high-risk items (partitioning)
   run in parallel with Phase 1 so go/no-go decisions are made early.
4. **Low effort, high value early** — cheap quick wins ship immediately and
   improve every subsequent development cycle.

## Phase 7 — Adaptive / Event-Driven Scheduler Wake (✅ Complete)

Must-ship (low latency is a primary project goal). Lands after Fuse (both
modify the scheduler entry point) and before ST-to-ST (which adds new CDC
wake signals).

| Item | Description | Effort | Status |
|------|-------------|--------|--------|
| WAKE-1 | `pg_notify('pgtrickle_wake', '')` from CDC triggers; 10 ms coalesce debounce; poll-based fallback when idle. Expected: median latency 515 ms → 15 ms (34×) | 2–3 wk | ✅ Done |

**Implementation notes:**
- **CDC triggers**: All trigger function builders (`build_row_trigger_fn_sql`,
  `build_stmt_trigger_fn_sql`, TRUNCATE trigger) now include
  `PERFORM pg_notify('pgtrickle_wake', '')` after every INSERT to the change
  buffer. PostgreSQL coalesces multiple NOTIFYs within a single transaction
  into one notification, so the overhead is negligible (~0.5 µs).
- **Scheduler LISTEN**: `pg_trickle_scheduler_main()` issues `LISTEN pgtrickle_wake`
  at startup when `pg_trickle.event_driven_wake = true` (default). PostgreSQL
  sets the scheduler's latch when a notification arrives (after the notifying
  transaction commits), causing `wait_latch()` to return immediately.
- **Debounce**: After an early latch return (event-driven wake), the scheduler
  does a secondary `wait_latch()` for `pg_trickle.wake_debounce_ms` (default 10 ms)
  to coalesce rapidly arriving notifications from bulk DML before starting
  the refresh tick.
- **GUC variables**: `pg_trickle.event_driven_wake` (bool, default `true`),
  `pg_trickle.wake_debounce_ms` (int, default `10`, range 1–5000).
- **Wake statistics**: The scheduler logs event-driven vs poll-based wake
  counts every 60 seconds at `LOG` level.
- **Poll fallback**: When event-driven wake is disabled or when no notifications
  arrive, the scheduler falls back to the existing `scheduler_interval_ms`
  poll-based wake.
- **Upgrade**: Existing trigger functions are rebuilt at runtime via
  `rebuild_cdc_trigger_function()` (uses `CREATE OR REPLACE FUNCTION`)
  on the first refresh cycle after upgrade.

**Exit criteria:**
- ✅ CDC triggers emit `pg_notify('pgtrickle_wake', '')` in all trigger modes
  (statement INSERT/UPDATE/DELETE, row-level, TRUNCATE)
- ✅ Scheduler LISTENs on `pgtrickle_wake` channel at startup
- ✅ 10 ms debounce coalesces rapid-fire notifications
- ✅ Poll-based fallback works when `event_driven_wake = off`
- ✅ E2E tests: trigger NOTIFY presence, event-driven latency, poll fallback
- ✅ `cargo fmt` clean; `cargo clippy --lib` zero warnings; 1475 unit tests pass

## Phase 10 — DAG Refresh Performance Improvements

Builds on the ST-to-ST differential infrastructure from Phase 8 and the
event-driven wake from Phase 7. Targets the scheduling and dispatch layer
to reduce end-to-end propagation latency by 30–50% for heterogeneous DAGs.

| Item | Description | Effort | Status |
|------|-------------|--------|--------|
| DAG-1 | **Intra-tick pipelining.** Within a single scheduler tick, begin processing a downstream ST as soon as all its specific upstream dependencies have completed — not when the entire topological level finishes. Requires per-ST completion tracking in the parallel dispatch loop and immediate enqueuing of newly-ready STs. Expected 30–50% latency reduction for DAGs with mixed-cost levels. | 2–3 wk | ✅ Done |
| DAG-2 | **Adaptive poll interval.** Replace the fixed 200 ms parallel dispatch poll with exponential backoff (20 ms → 200 ms), resetting on worker completion. Makes parallel mode competitive with CALCULATED for cheap refreshes. Alternative: `WaitLatch` with shared-memory completion flags. | 1–2 wk | ✅ Done |
| DAG-3 | **Delta amplification detection.** Track input→output delta ratio per hop via `pgt_refresh_history`. When a join ST amplifies delta beyond a configurable threshold (e.g., output > 100× input), emit a performance WARNING and optionally fall back to FULL for that hop. Expose amplification metrics in `explain_st()`. | 3–5d | ✅ Done |
| DAG-4 | **ST buffer bypass for single-consumer CALCULATED chains.** For ST dependencies with exactly one downstream consumer refreshing in the same tick, pass the delta in-memory instead of writing/reading from the `changes_pgt_` buffer table. Eliminates 2× SPI DML per hop. | 3–4 wk | ✅ Done |
| DAG-5 | **ST buffer batch coalescing.** Apply net-effect computation to ST change buffers before downstream reads — cancel INSERT/DELETE pairs for the same `__pgt_row_id` that accumulate between reads during rapid-fire upstream refreshes. | 1–2 wk | ✅ Done |

**Implementation order:** DAG-3 (lowest effort, immediate value) → DAG-2
(low-medium effort, unblocks parallel competitiveness) → DAG-1 (highest
impact, medium effort) → DAG-5 (leverages existing net-effect code) → DAG-4
(highest effort, requires in-memory delta plumbing).

**Implementation notes (DAG-3):**
- `pg_trickle.delta_amplification_threshold` GUC added to `config.rs` (float,
  default `100.0`, range 0.0–100,000). Set to `0.0` to disable detection.
- `compute_amplification_ratio(input_delta, output_delta) -> f64` pure-logic
  helper in `refresh.rs` — returns `output / input` (or `0.0` when input ≤ 0).
- `should_warn_amplification(input, output, threshold) -> bool` pure-logic
  predicate — returns `true` only when threshold > 0, input > 0, and
  ratio exceeds threshold.
- After each DIFFERENTIAL refresh MERGE completes, the amplification ratio
  is checked against the GUC threshold. On exceedance a `WARNING` is emitted
  with the ST name, input/output counts, computed ratio, and tuning hint.
- `explain_st()` now returns an `amplification_stats` property with JSON:
  `{"samples":N,"min":...,"max":...,"avg":...,"latest":...,"threshold":...}`
  computed from the last 20 DIFFERENTIAL refreshes in `pgt_refresh_history`.
- 14 unit tests: 6 for `compute_amplification_ratio`, 8 for
  `should_warn_amplification`, covering edge cases (zero input, disabled
  threshold, negative values, boundary). 1501 unit tests pass.

**Implementation notes (DAG-2):**
- Replaced the fixed `min(scheduler_interval_ms, 200)` poll cap with
  exponential backoff: 20ms → 40ms → 80ms → 160ms → 200ms (cap).
- `compute_adaptive_poll_ms(current, had_completion, has_inflight, base) -> u64`
  pure-logic helper in `scheduler.rs` — returns `ADAPTIVE_POLL_MIN_MS` (20ms)
  on worker completion, doubles the current interval when no completion with
  workers in-flight, or returns the full base interval when idle.
- Constants `ADAPTIVE_POLL_MIN_MS = 20` and `ADAPTIVE_POLL_MAX_MS = 200`
  define the backoff range.
- `ParallelDispatchState` extended with `adaptive_poll_ms` and
  `completions_this_tick` fields. Completions are tracked in
  `parallel_dispatch_tick` Step 1 when a terminal job is detected.
- The scheduler main loop computes the next poll interval via
  `compute_adaptive_poll_ms()` at the top of each iteration, then
  clears `completions_this_tick` for the next tick.
- 8 unit tests covering: no-inflight returns base, completion resets to min,
  doubling per tick, cap at max, full backoff sequence, transition from
  inflight to idle, and initial state validation. 1509 unit tests pass.

**Implementation notes (DAG-1):**
- The Phase 4 parallel dispatch architecture already achieves intra-tick
  pipelining by design. Step 1 of `parallel_dispatch_tick()` immediately
  decrements `remaining_upstreams` for downstream units when an upstream
  completes. Step 2 then picks up any freshly-ready units (remaining == 0)
  in the same tick, and Step 3 dispatches them within quota.
- There is no level barrier: in a mixed-cost DAG where level 0 has both
  A (fast) and B (slow), downstream C (depends only on A) is dispatched
  as soon as A completes, without waiting for B.
- 3 validation tests added: cascade pipeline state tracking (A → B → C),
  mixed-cost levels with no level barrier, and wave reset correctness
  for multi-level cascades. 1521 unit tests pass.

**Implementation notes (DAG-5):**
- `compact_st_change_buffer(schema, pgt_id, prev_lsn, new_lsn)` in `cdc.rs`
  applies the same window-function net-effect compaction to ST change buffers
  (`changes_pgt_{id}`) as `compact_change_buffer()` does for base-table
  buffers. Removes:
  - Net-zero groups: first_action='I' and last_action='D' for same pk_hash.
  - Intermediate rows: keeps only first and last per pk_hash group.
- Pure-logic helpers for testability:
  - `build_st_compact_sql()` generates the compaction DELETE SQL.
  - `compact_st_advisory_lock_key()` computes the lock key (namespace
    0x5047_5500, distinct from base-table 0x5047_5400).
- Uses the same `pg_trickle.compact_threshold` GUC as base-table compaction.
  Skips when threshold ≤ 0 or pending count ≤ threshold.
- Integrated in `execute_differential_refresh()` right after C-4 base-table
  compaction and before the decision query at E-1.
- 9 unit tests: SQL generation (table name, LSN range, schema, pk_hash
  partitioning, net-zero removal, intermediate removal) and advisory lock
  key tests (namespace isolation, pgt_id embedding, uniqueness). 1521 unit
  tests pass.

**Implementation notes (DAG-4):**
- New `ExecutionUnitKind::FusedChain` variant in `dag.rs` for chains of
  Singleton CALCULATED stream tables where each upstream has exactly one
  downstream consumer.
- `find_fusable_chains(eu_dag, resolve_mode)` pure-logic function detects
  maximal chains (length ≥ 2) by greedy forward walk from eligible
  Singleton units. Conditions: both upstream and downstream are Singletons
  with DIFFERENTIAL refresh mode, and each upstream has exactly one
  downstream unit in the EU DAG.
- `fuse_chain()` method on `ExecutionUnitDag` merges a chain of Singleton
  unit IDs into a single FusedChain unit, rewiring external dependency
  edges and updating the `pgt_to_unit` mapping.
- Post-processing step added in `build_from_st_dag()` Step 5: detect
  fusable chains and merge them after building inter-unit edges.
- `execute_worker_fused_chain(job)` in `scheduler.rs` runs all chain
  members sequentially in the same background worker, similar to
  `execute_worker_atomic_group()` but with bypass temp table capture
  between consecutive members.
- `capture_delta_to_bypass_table(st, user_cols)` in `refresh.rs` creates
  `pg_temp.__pgt_bypass_{pgt_id}` (ON COMMIT DROP) with the same schema
  as the persistent `changes_pgt_` buffer. Inserts I/D rows from the
  delta temp table and registers the mapping in a thread-local
  `ST_BYPASS_TABLES: HashMap<i64, String>`.
- `build_bypass_capture_sql()` pure-logic helper for testable SQL gen.
- Thread-local bypass mapping propagated to `DiffContext.st_bypass_tables`
  via `dvm/mod.rs`. When bypass tables are active, the delta SQL cache is
  bypassed (table names are baked into cached templates).
- `diff_scan_change_buffer()` in `dvm/operators/scan.rs` checks
  `ctx.st_bypass_tables` before constructing the change table name.
  When a bypass entry exists for an ST source pgt_id, reads from the temp
  table instead of the persistent buffer.
- `pgt_scheduler_jobs.unit_kind` CHECK constraint widened to include
  `'fused_chain'`; upgrade SQL added to `pg_trickle--0.10.0--0.11.0.sql`.
- 11 unit tests: chain detection (simple pair, triple, fan-out not
  eligible, non-differential not eligible, independent singletons),
  edge preservation, stable key, kind display, is_calculated helper,
  summary format, topological order. Plus 4 refresh.rs tests for bypass
  SQL generation and thread-local lifecycle. 1532 unit tests pass.

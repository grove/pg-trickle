# PLAN_0_11_0.md — v0.11.0 Implementation Order

**Milestone:** v0.11.0 — Partitioned Stream Tables, Prometheus & Grafana Observability, Safety Hardening & Correctness
**Last updated:** 2026-03-26
**Phase 1 status:** ✅ Complete (merged PR #279, 2026-03-25)
**Phase 2 status:** ✅ Complete (PR #TBD, 2026-03-26)

This document defines the recommended implementation order for all v0.11.0
roadmap items. Sequencing is driven by:

1. **Correctness and safety first** — panics and silent failures must be
   eliminated before adding new code paths.
2. **Dependency order** — later phases build on infrastructure introduced by
   earlier ones (e.g. WB-1 must land before ST-to-ST; WAKE-1 before C3-1).
3. **Risk front-loading** — design spikes for high-risk items (partitioning)
   run in parallel with Phase 1 so go/no-go decisions are made early.
4. **Low effort, high value early** — cheap quick wins ship immediately and
   improve every subsequent development cycle.

## Phase 7 — Adaptive / Event-Driven Scheduler Wake (~2–3 weeks)

Must-ship (low latency is a primary project goal). Lands after Fuse (both
modify the scheduler entry point) and before ST-to-ST (which adds new CDC
wake signals).

| Item | Description | Effort |
|------|-------------|--------|
| WAKE-1 | `pg_notify('pgtrickle_wake', '')` from CDC triggers; 10 ms coalesce debounce; poll-based fallback when idle. Expected: median latency 515 ms → 15 ms (34×) | 2–3 wk |

**Exit:** Median end-to-end latency ≤ 20 ms for low-volume workloads.
Event-driven and poll-based paths both covered by E2E tests.

## Phase 8 — Differential ST-to-ST Refresh (~4.5–6.5 weeks)

The largest item. Builds on CDC infrastructure (WB-1), scheduler stability
(WAKE-1, FUSE), and catalog work (G12-ERM). Implement in strict sub-phase
order — each sub-phase is a reviewable unit.

| Sub-phase | Item | Description | Effort |
|-----------|------|-------------|--------|
| 8.1 | ST-ST-1 | Change buffer infrastructure: `create_st_change_buffer_table()` / `drop_st_change_buffer_table()` in `cdc.rs`; lifecycle hooks in `api.rs`; auto-create on upgrade | 1 wk |
| 8.2 | ST-ST-2 | Delta capture — DIFFERENTIAL path: `MERGE … RETURNING` to atomically apply MERGE and write delta to `changes_pgt_{id}` | 1–2 wk |
| 8.3 | ST-ST-3 | Delta capture — FULL path: pre/post snapshot diff; downstream STs always receive I/D delta regardless of upstream mode | 1 wk |
| 8.4 | ST-ST-4 | DVM scan operator for ST sources: read from `changes_pgt_{id}` instead of `changes_{oid}`; `pgt_`-prefixed LSN tokens; extend frontier computation | 1–2 wk |
| 8.5 | ST-ST-5 | Scheduler integration: rewrite `has_stream_table_source_changes()` to check ST buffer; remove `has_stream_table_changes → Full` override when buffer exists | 0.5 wk |
| 8.6 | ST-ST-6 | Cleanup, lifecycle, tests: extend `drain_pending_cleanups()`; hook `hooks.rs` for DDL; unit + integration + E2E tests (3-level chain, DROP-middle fallback, schema change buffer recreation, min-frontier cleanup) | 1 wk |

**Exit:** ST → ST → ST 3-level chain refreshes differentially. Pre/post
snapshot path means no cascading FULL. DROP-middle-ST falls back gracefully.

effort.

# ADR-003: EC-01 Join-Correctness Invariant and the PHD1 Cleanup Pass

| Field        | Value    |

| **Updated**  | 2026-07-01 (v0.61.0 — promoted to explicit ADR; PHD1 ctid invariant documented) |
| **Deciders** | pg_trickle core team |

---

## Context

The standard incremental join rule for a two-source join R ⋈ S is:

```
Δ(R ⋈ S) = ΔR ⋈ S₀  ∪  R₀ ⋈ ΔS
```

where R₀ and S₀ are the *pre-delta snapshots* of R and S.

For **LEFT OUTER JOINs**, the delta rule is more complex because a row from R
that previously had no match in S (producing a NULL-padded output row) may now
have a match after ΔS is applied, or vice versa.  The naïve delta rule can
produce **phantom rows** — rows that appear in the stream table after a refresh
but should have been deleted.

The phantom-row problem was identified empirically in v0.37.x and fixed in
v0.38.0.

---

## Decision

### EC-01: R₀ Snapshot-Splitting (v0.38.0)

The outer-join delta rules in `src/dvm/operators/left_join.rs` (and related
files) use **two separate snapshots of the left source R**:

- `R₀_before` — state of R *before* ΔR is applied (used to cancel old output rows)
- `R₀_after` — state of R *after* ΔR is applied (used to generate new output rows)

This ensures that when a left row moves from "no match → has match" or
"has match → no match", both the old phantom-padded row and the new correctly-joined
row are accounted for in a single delta application.

### PHD1: Cross-Cycle Phantom Cleanup

Even with EC-01 snapshot-splitting, residual phantom rows can accumulate under
specific patterns:

1. A left-outer-join stream table is refreshed while a concurrent transaction
   holds a snapshot that predates the source write.
2. The snapshot isolation boundary causes ΔS to appear empty for this refresh
   cycle, so the matched row is not yet propagated.
3. The next refresh cycle processes ΔS correctly, but does not cancel the
   phantom row created in the previous cycle.

The **PHD1 cleanup pass** (`src/refresh/phd1.rs`) runs after every differential
refresh of an outer-join stream table.  It:

1. Identifies rows in the stream table with multiplicity > 1 or < 0.
2. Uses `ctid`-based deletion within a single transaction to correct multiplicities.
3. Re-inserts any net-positive rows that were over-deleted.

---

## ctid Invariant

The PHD1 pass uses `ctid` values captured in a CTE to identify target rows for
deletion.  This is safe because:

> **INVARIANT**: The `ctid` values captured by the preceding CTE are stable
> within this transaction's snapshot (no concurrent VACUUM or HOT chain
> updates).  Do NOT split this DELETE into a separate transaction or
> sub-transaction — if the snapshot isolation guarantee is lost, use
> `__pgt_row_id`-based deletion instead.

This invariant is documented in code at `src/refresh/phd1.rs` (COR-7, v0.61.0).

---

## Conditions Under Which PHD1 Must Run

The PHD1 pass is **required** when ALL of the following are true:

1. The stream table's defining query contains at least one `LEFT JOIN`,
   `RIGHT JOIN`, or `FULL OUTER JOIN`.
2. The stream table is refreshed in `DIFFERENTIAL` mode.
3. The refresh cycle captured at least one change (ΔR ≠ ∅ or ΔS ≠ ∅).

The PHD1 pass is **skipped** when:
- The stream table uses only `INNER JOIN` (no phantom-row risk).
- The stream table is refreshed in `FULL` mode (full recompute corrects any state).
- No changes were captured in this cycle (no delta to apply).

---

## Guarantees Provided by PHD1

After PHD1 completes:

1. Every row in the stream table has multiplicity exactly 1.
2. Rows absent from the stream table have multiplicity 0.
3. No phantom NULL-padded rows remain from previous cycles.

These guarantees hold **within a single-node PostgreSQL instance** under
standard Read Committed or Repeatable Read isolation.  Serializable isolation
is not required by pg_trickle, but is compatible.

---

## Trade-offs

| Concern | Mitigation |
|---------|-----------|
| PHD1 adds latency to every differential refresh of outer-join queries | Amortised cost is proportional to phantom row count (typically zero in steady state) |
| ctid-based deletion is not safe across vacuums | PHD1 runs within a single transaction; VACUUM cannot run concurrently on the same table within that transaction |
| PHD1 may interact with row-level security | RLS is bypassed for the cleanup because pg_trickle functions run as SECURITY DEFINER |

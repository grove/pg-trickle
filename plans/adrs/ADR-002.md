# ADR-002: Z-Set / Multiset Formalism for Differential Dataflow

|--------------|----------|

---

## Context

The pg_trickle differential view maintenance (DVM) engine computes *deltas*
— the minimal set of changes to apply to a stream table after a source change.
To do this, the engine needs a mathematical formalism for representing
"collections of rows with changes".

Two candidate formalisms were evaluated:

1. **True Multiset (bag)** — a collection where each row can appear zero or
   more times; multiplicities are non-negative integers.
2. **Z-Set** — a collection where each row has an integer multiplicity that can
   be negative; negative multiplicity represents "remove this row".

---

## Decision

The **differential engine uses Z-sets (signed integer multiplicities)**.

In practice, all stream-table rows have multiplicity +1 (present) or 0
(absent).  During a delta computation, intermediate results may transiently
carry multiplicity −1 (deletion) or +2 (double-add before deduplication).

---

## Rationale

### SQL Deletes Require Negative Multiplicities

SQL `DELETE` and `UPDATE` semantics require "undo" of a previously added row.
A true multiset with only non-negative multiplicities cannot represent "subtract
this row from the collection".

Z-sets handle deletions naturally: inserting a row with multiplicity −1 cancels
a previous +1, yielding net multiplicity 0 (absent).

### Compositional Operator Algebra

Z-set operators compose cleanly:

- `σ` (Filter):  `Δ(σ(R)) = σ(ΔR)` — apply filter to the delta.
- `π` (Project): `Δ(π(R)) = π(ΔR)` — apply projection to the delta.
- `⋈` (Join):    `Δ(R ⋈ S) = ΔR ⋈ S₀ ∪ R₀ ⋈ ΔS` — incremental join rule.
- `γ` (GroupBy): requires separate handling for count/sum changes.
- `∪` (Union):   `Δ(R ∪ S) = ΔR ∪ ΔS`.

Each operator file in `src/dvm/operators/` implements the corresponding Z-set
delta rule for its SQL construct.

### Correctness Invariant

After each complete refresh cycle, the sum of multiplicities for any given row
must be either:
- **0** — row is absent from the stream table, OR
- **1** — row is present exactly once.

A multiplicity of +2 or −1 at the end of a cycle indicates a bug in the delta
rules.  The `src/refresh/phd1.rs` (PHD1) cleanup pass detects and corrects
phantom rows that can accumulate from outer-join delta rules under certain
concurrency patterns; see ADR-003.

---

## Implications for Operator Correctness

- All 22 operator files in `src/dvm/operators/` must maintain the Z-set
  invariant throughout their delta computations.
- The `DiffContext` (`src/dvm/diff.rs`) accumulates CTEs that collectively
  compute the delta in Z-set terms.
- Snapshot CTEs (the `R₀` captures of source-table state *before* the delta)
  are shared across operators to avoid redundant scans; see also COR-8
  (snapshot cache secondary equality check, v0.61.0).
- The `cte_counter` in `DiffContext` is reset at the start of each
  `differentiate()` call (COR-9, v0.61.0) to prevent stale CTE name reuse
  across invocations.

---

## Alternatives Considered

### CRDT-Based Formalism

Conflict-free Replicated Data Types (CRDTs) were considered for their
strong eventual consistency properties.  Rejected because:
- CRDTs are designed for distributed systems with network partitions;
  pg_trickle operates within a single PostgreSQL instance.
- CRDT merge semantics do not map cleanly to SQL outer-join semantics.

### Event Sourcing (Append-Only Log + Replay)

Storing all changes as an immutable log and replaying from scratch for each
refresh was considered.  Rejected because:
- Replay is O(total history), not O(delta) — violates the low-latency goal.
- This is effectively what FULL refresh mode does; DIFFERENTIAL exists precisely
  to avoid it.

# ADR-001: Trigger-Based CDC over Logical Replication

| Field        | Value    |
|--------------|----------|
| **Status**   | Accepted |
| **Date**     | 2024-11-01 |
| **Updated**  | 2026-07-01 (v0.61.0 — promoted from implied decision to explicit ADR) |
| **Deciders** | pg_trickle core team |

---

## Context

pg_trickle needs to capture every row-level change (INSERT, UPDATE, DELETE,
TRUNCATE) on source tables and store the delta in change-buffer tables so the
differential refresh engine can apply incremental updates to stream tables.

Two primary mechanisms are available in PostgreSQL for this purpose:

1. **Row/statement-level AFTER triggers** — fire synchronously within the
   source transaction and write change records to buffer tables.
2. **Logical replication / WAL decoding** — asynchronous CDC via a replication
   slot; a background process reads WAL and applies deltas out-of-band.

---

## Decision

The **default CDC path uses AFTER row-level and statement-level triggers**.
Logical replication (WAL-based CDC) is provided as an opt-in alternative via
`REFRESH MODE WAL`.

---

## Rationale

### Single-Transaction Atomicity

With trigger-based CDC, the change record is written to the buffer table
**within the same transaction** as the source write.  If the source transaction
rolls back, the trigger's changes roll back with it — no compensating logic
required.

With logical replication the change is captured asynchronously from WAL *after*
the source transaction commits.  A crash or failover between the commit and WAL
decoding can result in a window where the stream table has not been updated.
While PostgreSQL replication is durable once the WAL record is flushed, the
additional moving parts (slot management, apply lag, slot lag accumulation)
increase operational complexity.

### Simpler Failure Modes

Trigger-based CDC fails loudly and synchronously.  A trigger error aborts the
source transaction, making it immediately visible to the application.

Logical replication failures (slot lag growing unbounded, replication slot
invalidation due to `max_replication_slots` exhaustion) are harder to detect and
can silently cause data divergence.

### No Replication Slot Lifecycle Management

Replication slots must be created, monitored, and dropped carefully.
An orphaned slot causes WAL accumulation that can fill disk.  Trigger-based CDC
requires no slot management.

### Statement-Level Trigger Optimization

For bulk operations (`INSERT INTO … SELECT`, `COPY`), pg_trickle installs a
**statement-level AFTER trigger** alongside the row-level trigger.  The
statement-level trigger captures the set of affected rows in a single pass,
avoiding O(n) per-row trigger overhead for large batches.

---

## Trade-offs Acknowledged

| Concern | Mitigation |
|---------|-----------|
| Higher write amplification (trigger overhead per DML) | Statement-level triggers for bulk writes; trigger skip for TRUNCATE → full refresh |
| Cannot capture changes from logical replication subscribers | WAL-based CDC (`REFRESH MODE WAL`) available as opt-in |
| Requires `BEFORE`/`AFTER` trigger support on the source table | All standard PostgreSQL tables support triggers; foreign tables do not |
| Not suitable for tables with extremely high write rates (>100k rows/s steady state) | WAL-based CDC available; cost model surfaces recommendation |

---

## Consequences

- Every source table tracked by pg_trickle has two triggers installed:
  `__pgt_row_trigger_<oid>` (row-level) and optionally a statement-level
  counterpart.
- Change buffers (`pgtrickle_changes.changes_<oid>`) are created and managed by
  the extension.
- See also: ADR-003 (EC-01 Join-Correctness Invariant) for how trigger-captured
  change records interact with the differential join delta rules.

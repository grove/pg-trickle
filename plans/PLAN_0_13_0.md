# PLAN_0_13_0.md — v0.13.0 Implementation Order

**Milestone:** v0.13.0 — Scalability Foundations, Partitioning Enhancements, MERGE Profiling & Multi-Tenant Scheduling
**Status:** 🚧 In Development
**Last updated:** 2026-03-28

This document defines the recommended implementation order for all v0.13.0
roadmap items. The scope is the set of phases deferred from v0.12.0 (Phases
5–8) plus three new phases added after the v0.12.0 scope was locked.

Sequencing is driven by:

1. **Dependency order** — A-2 (columnar tracking) must precede D-4 (shared
   buffers); TPC-H benchmark harness (Phase 10) should land before the
   scalability work so that performance gains can be measured objectively.
2. **Risk front-loading** — Phase 10 (TPC-H harness) and Phase 11 (SQL
   cleanup) are cheap, zero-risk, and provide immediate value; ship them
   in Phases 1–2 so the rest of the milestone has a measurement baseline.
3. **Correctness-sensitive phases last** — D-4 multi-frontier cleanup is
   the hardest correctness hazard in this release. Gate it behind A-2 and
   behind a dedicated property-based test harness.
4. **Low effort, high value early** — MERGE profiling, dbt macro updates,
   and SQL cleanup ship early; they are independent of the heavy structural
   work and de-risk the milestone exit criteria.

## Priority Tiers (Scope Risk)

Total estimated scope is ~15–23 weeks across all nine phases. If the
milestone runs long, phases are deferrable in the order below. A release
can be cut with only the P0 phases complete.

| Tier | Phases | Contents | Action if time is short |
|------|--------|----------|-------------------------|
| **P0 — Must ship** | 1, 6, 7 | TPC-H harness, A-2 columnar tracking, D-4 shared buffers | Block release until done |
| **P1 — Target** | 3, 5, 8, 9 | MERGE profiling, partitioning enhancements, PERF-2 auto-partitioning, multi-tenant scheduler isolation | Ship if ready; defer individual items to v0.14.0 if not |
| **P2 — Can slip** | 2, 4 | SQL coverage cleanup, dbt macro updates | Defer entire phase to v0.14.0 as a unit |

> **Slip rule:** Never partially ship Phase 7 (D-4). It depends on Phase 6
> (A-2) and a partial shared-buffer migration is worse than no migration.
> If Phase 6 is deferred, Phase 7 must be deferred with it.

## Phase 6 — Columnar Change Tracking (A-2)

**Goal:** Eliminate wasted delta processing for wide tables. When a row is
updated but only projected columns changed, the current CDC path still emits a
full row delta. A-2 introduces a per-column bitmask that lets the Delta Scan
operator skip rows where no referenced column changed — the foundation for the
shared-buffer work in Phase 7.

> ⚠️ **Implement before Phase 7.** D-4 (shared buffers) builds on the
> per-column tracking infrastructure added here. The column-superset mode
> required by D-4 is a natural extension of the A-2 bitmask.

| Item | Description | Effort | Ref |
|------|-------------|--------|-----|
| A-2 | **Columnar Change Tracking.** Per-column bitmask in CDC triggers; extend the existing `changed_cols VARBIT` column (added in v0.11.0 WB-1) to carry a full per-column presence flag; skip rows where no projected column is set in the bitmask; add a lightweight UPDATE-only delta path when only projected columns changed. Returns `HashMap<u32, Vec<String>>` mapping table OID → key column names. Uses three helper methods: `collect_key_column_names()` (recursive tree walk), `collect_column_names_from_expr()` (expression extraction), `collect_scan_alias_map()` (alias→OID resolution from Scan nodes). `ParseResult::source_key_columns_used()` delegates to OpTree + CTE bodies, merging results.

- **Key-only bitmask (`compute_varbit_key_cols_mask()`):** New function in `src/dvm/operators/scan.rs` that computes a VARBIT mask covering only key columns. Returns `None` when all columns are keys (no optimization possible) or no columns are keys.
- **`__pgt_key_changed` annotation:** When key columns are a strict subset of all columns, scan delta CTEs emit `__pgt_key_changed BOOLEAN` column. For single-change PKs: `CASE WHEN changed_cols & key_mask = zero_mask THEN FALSE ELSE TRUE END`. For multi-change PKs: `bool_or()` window function ensures TRUE if ANY change in the PK's batch touched a key column. Propagated through raw_cte (via `SELECT *`) and into both merge_safe_dedup and standard delta CTE outputs.
- **E2E tests:** 4 tests in `tests/e2e_columnar_tracking_tests.rs`: wide-table unreferenced update skipped, referenced value update applied, key column update moves group, mixed referenced/unreferenced updates.
- **Unit tests:** 5 tests for `source_key_columns_used()` (aggregate/group_by, scan_only, inner_join, filter_where, aggregate_with_filter) + 8 tests for `compute_varbit_key_cols_mask()` and key annotation (basic, multiple_keys, all_keys, no_keys, no_match, annotation_present, annotation_absent) + 7 tests for P5 value-only optimization (SUM with/without value-only, COUNT* ignores value-only, COUNT(col) ignores value-only, P5 with key cols, P5 without key cols, P5 all-keys).
- **P5 aggregate value-only optimization:** In `generate_direct_agg_delta()` (src/dvm/operators/aggregate.rs), when key mask is available via `compute_varbit_key_cols_mask()`, extends the LATERAL VALUES expansion with a third 'V' (value-only) row. For value-only UPDATEs (`changed_cols & key_mask = zero`), only the 'V' row passes — carrying `(new_col - old_col)` as net correction for SUM, and contributing 0 to counts. Standard I+D sides are suppressed for these rows. The WHERE clause routes: `v.side = 'I'` for non-value-only inserts/updates, `v.side = 'D'` for non-value-only deletes/updates, `v.side = 'V'` for value-only updates only. SUM expressions include 'V' via `v.side IN ('I', 'V')`. COUNT/COUNT(*) expressions only match 'I'/'D', correctly reflecting that value-only updates don't change group membership.

## Phase 7 — Shared Change Buffers (D-4)

**Goal:** Replace per-ST change buffer tables with a single shared buffer per
source table, shared across all dependent stream tables. Reduces I/O
multiplication in deployments where dozens of stream tables read from the same
source. Multi-frontier cleanup coordination is the hardest correctness hazard
in this release.

> ⚠️ **Requires Phase 6.** D-4 builds on A-2's column-superset tracking
> infrastructure. Do not start Phase 7 until Phase 6 is merged and green in CI.

> ⚠️ **Multi-frontier cleanup correctness.** Each downstream ST advances its
> own frontier independently. The shared buffer must not be truncated past the
> minimum frontier across all consumers. Always use `MIN(consumer_frontier)` in
> the cleanup query — never `MAX`.

| Item | Description | Effort | Ref |
|------|-------------|--------|-----|
| D-4 | **Shared Change Buffers.** Single buffer per source table shared across all dependent STs; multi-frontier cleanup coordination via per-consumer frontier tracking; static-superset column mode for initial implementation (shared buffer captures all columns referenced by any consumer). | ~3–4 wk | [PLAN_NEW_STUFF.md §D-4](performance/PLAN_NEW_STUFF.md) |

**Implementation sub-tasks:**

## Phase 9 — Multi-Tenant Scheduler Isolation ✅ Complete

**Goal:** As deployments grow past 10 databases on a single cluster, all
schedulers compete for the same global background-worker pool. One busy
database can starve the others. Phase 9 delivers per-database worker quotas
and a priority queue so critical databases always get workers.

> ⚠️ C-3 depends on C-1 (tiered scheduling) for Hot/Warm/Cold classification.
> If C-1 is not fully ready, fall back to `IMMEDIATE > all-other` ordering

> follow-on.

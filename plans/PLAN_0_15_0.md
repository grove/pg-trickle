# PLAN_0_15_0.md — v0.15.0 Implementation Order

**Milestone:** v0.15.0 — External Test Suites & Integration
**Status:** 🔄 In Progress
**Last updated:** 2026-04-03

This document defines the recommended implementation order for all v0.15.0
roadmap items.

Sequencing is driven by:

1. **Dependency order** — G13-PRF (parser modularization) is a prerequisite
   for PG backward compatibility in v0.16.0; front-load it if possible.
2. **Risk front-loading** — Phase 1 (TS3 + I2) ships independently
   of the two large structural items (G13-PRF, WM-7).
3. **Independent features early** — TS3, I2, I3, G15-BC, and PH-E1 are structurally
   independent; any can slip without blocking the others.
4. **Large refactors later** — G13-PRF is ~3–4 weeks of careful work; it
   should not block the release if deadlines tighten.

## Phase 3 — Watermark Hold-Back Mode (WM-7)

**Goal:** Detect stuck watermarks, pause affected stream tables, emit alerts,
and auto-resume when watermarks advance.

- [ ] Add `watermark_holdback_timeout` GUC to `src/config.rs`
- [ ] Implement stuck watermark detection in `src/wal_decoder.rs` or scheduler
- [ ] Pause downstream gated stream tables when stuck detected
- [ ] Emit `pgtrickle_alert` NOTIFY with category `watermark_stuck`
- [ ] Implement auto-resume when watermark advances
- [ ] Write integration tests: stuck / unstuck / re-gating scenarios
- [ ] Update `sql/pg_trickle--0.14.0--0.15.0.sql` with DDL changes
- [ ] Update `sql/archive/pg_trickle--0.15.0.sql`
- [ ] Document in `docs/SQL_REFERENCE.md` and `docs/CONFIGURATION.md`

**Ref:** [PLAN_WATERMARK_GATING.md §4.1](plans/sql/PLAN_WATERMARK_GATING.md)

## Phase 4 — Delta Cost Estimation (PH-E1)

**Goal:** Before executing delta SQL, estimate intermediate cardinality and
downgrade to FULL refresh with `NOTICE` if the estimate exceeds budget.
Prevents OOM and runaway temp-file spills on unexpected large deltas.

- [ ] Add `pg_trickle.max_delta_work_mem_mb` GUC to `src/config.rs` (default: 2× `work_mem`)
- [ ] Implement cardinality estimator: change buffer row count × join fan-out heuristic in `src/refresh.rs`
- [ ] Wire into differential refresh path: check estimate, downgrade to FULL + emit `NOTICE` if exceeded
- [ ] Write integration tests: normal delta passes through; oversized delta triggers FULL downgrade
- [ ] Document `max_delta_work_mem_mb` in `docs/CONFIGURATION.md`

**Ref:** [PLAN_PERFORMANCE_PART_9.md §Phase E](plans/performance/PLAN_PERFORMANCE_PART_9.md)

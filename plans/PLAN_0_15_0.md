# PLAN_0_15_0.md — v0.15.0 Implementation Order

**Milestone:** v0.15.0 — External Test Suites & Integration
**Status:** 🔄 In Progress
**Last updated:** 2026-04-03

This document defines the recommended implementation order for all v0.15.0
roadmap items.

Sequencing is driven by:

1. **Dependency order** — G13-PRF (parser modularization) is a prerequisite
   for PG backward compatibility in v0.16.0; front-load it if possible.
2. **Risk front-loading** — Phase 1 (TS3 + I2) ships independently
   of the two large structural items (G13-PRF, WM-7).
3. **Independent features early** — TS3, I2, I3, G15-BC, PH-E1, PH-D2, TRUNC-1,
   VOL-1, EC-01, STST-3, CIRC-IMM, E4, and E5 are structurally independent; any
   can slip without blocking the others.
4. **Large refactors later** — G13-PRF is ~3–4 weeks of careful work; it
   should not block the release if deadlines tighten.
5. **Research spike early** — G14-SHC-SPIKE informs v0.16.0 planning; do it
   early so the RFC is available before parser modularization completes.

**Goal:** Close the correctness gap where TRUNCATE on a source table in

- [x] Add a DDL event trigger or statement-level trigger that catches TRUNCATE on source tables (implemented via CDC AFTER TRUNCATE trigger in cdc.rs)
- [x] Mark downstream STs for `needs_reinit` when TRUNCATE detected (implemented in refresh.rs — detects 'T' marker at refresh time and falls back to FULL)
- [ ] Write E2E test: TRUNCATE source → verify ST marked for reinit → next refresh reinitializes

### EC-01: JOIN Key Change + DELETE Compensating Anti-Join

**Goal:** Close the G1.1 correctness gap where a simultaneous JOIN key update

- [ ] Wire into both `diff_inner_join` and `diff_left_join` paths
- [ ] Write E2E test: UPDATE join key + DELETE join partner in same cycle → verify stream table correct
- [ ] Write boundary tests: key change without partner delete (no-op), partner delete without key change (handled by existing delta)
- [ ] Update `docs/SQL_REFERENCE.md` to remove the documented limitation (currently under known edge cases)

## Phase 5 — Delta Cost & Spill Estimation (PH-E1 + PH-E2)

**Goal:** Before executing delta SQL, estimate intermediate cardinality and
downgrade to FULL refresh with `NOTICE` if the estimate exceeds budget.
Prevents OOM and runaway temp-file spills on unexpected large deltas.

- [ ] Add `pg_trickle.max_delta_work_mem_mb` GUC to `src/config.rs` (default: 2× `work_mem`)
- [ ] Implement cardinality estimator: change buffer row count × join fan-out heuristic in `src/refresh.rs`
- [ ] Wire into differential refresh path: check estimate, downgrade to FULL + emit `NOTICE` if exceeded
- [ ] Write integration tests: normal delta passes through; oversized delta triggers FULL downgrade
- [ ] Document `max_delta_work_mem_mb` in `docs/CONFIGURATION.md`

### PH-E2: Spill-Aware Refresh

**Goal:** Monitor `temp_bytes` after each refresh cycle and auto-adjust if
spill is excessive. Companion to PH-E1 (depends on PH-E1 being done first).

- [ ] Query `pg_stat_statements` after each refresh to get `temp_bytes`
- [ ] Track per-ST spill history (rolling window of last N refreshes)
- [ ] If spill exceeds threshold 3 consecutive times: auto-increase per-ST `work_mem` or switch to FULL
- [ ] Expose `spill_history` in `explain_st()` output
- [ ] Write integration tests: simulate spill → observe auto-adjustment
- [ ] Document in `docs/CONFIGURATION.md`

**Ref:** [PLAN_PERFORMANCE_PART_9.md §Phase E](plans/performance/PLAN_PERFORMANCE_PART_9.md)

## Phase 12 — Operational Quick Wins (G8.1, EXPL-ENH, R4)

### G8.1: Cross-Session MERGE Cache Invalidation

**Goal:** Prevent stale MERGE templates when one session alters a stream
table's query while another session has the old template cached.

- [ ] Add `catalog_version` column to `pgt_stream_tables` (INT, default 1)
- [ ] Bump `catalog_version` on ALTER QUERY, DROP, and reinit
- [ ] Before each refresh, compare cached version to catalog; regenerate template on mismatch
- [ ] Write E2E test: session A alters query, session B refreshes → picks up new template without error
- [ ] Write boundary test: version counter survives across connection pool cycling

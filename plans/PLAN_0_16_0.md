# PLAN_0_16_0.md ‚Äî v0.16.0 Implementation Order

**Milestone:** v0.16.0 ‚Äî Performance & Refresh Optimization
**Status:** üöß In Progress
**Last updated:** 2026-04-05

This document defines the recommended implementation order for all v0.16.0
roadmap items.

Sequencing is driven by:

1. **Quick wins first** ‚Äî C2-BUG (`resume_stream_table()` missing), ERR-REF,
   GUC-DEFAULTS, BUF-LIMIT are all self-contained and low-risk; ship them first to
   avoid blocking the release on large items.
2. **Test coverage before optimization** ‚Äî TG2 gaps (window, join, equivalence)
   must exist *before* landing PH-D1, B-1, A-3-AO, B-2, and C-4 so regressions
   surface immediately.
3. **Compaction before predicate pushdown** ‚Äî C-4 (change buffer compaction)
   reduces input volume for all downstream paths; implement before B-2 so pushdown
   benchmarks reflect realistic buffer sizes.
4. **Template cache after MERGE alternatives** ‚Äî G14-SHC benefits from stable MERGE
   templates; implement after PH-D1/B-1 decide the final template shape.
5. **PG 19 compat gated on beta availability** ‚Äî A3 work starts with pgrx bump and
   API audit; full CI validation waits for PG 19 beta.
6. **Benchmark CI last** ‚Äî BENCH-CI closes the release by gating regressions against
   the completed optimization set.

## Phase Details

### Phase 1 ‚Äî Quick Wins

Self-contained, low-risk items. All can be done in parallel.

#### C2-BUG ‚Äî `resume_stream_table()` is missing (P0 bug)

The function is referenced in error messages when a stream table is in
`SUSPENDED` status but does not exist as a callable SQL function. Users hitting
`SUSPENDED` state have no way to recover without direct catalog manipulation.

- Implement `pgtrickle.resume_stream_table(name TEXT)` in `src/api.rs`
- Clears `SUSPENDED` status; re-enqueues the ST for the next scheduler tick
- E2E test: blow fuse ‚Üí `SUSPENDED` ‚Üí `resume_stream_table()` ‚Üí scheduler resumes
- SQL migration in `sql/pg_trickle--0.15.0--0.16.0.sql`

#### ERR-REF ‚Äî Error reference documentation

Document all `PgTrickleError` variants with meaning, causes, and fixes.

- Create `docs/ERRORS.md` listing all 19 variants
- For each: one-sentence description, common trigger scenario, suggested fix,
  and example error message
- Cross-link from `docs/FAQ.md` and `docs/SQL_REFERENCE.md`
- Example: `UnsupportedOperator` ‚Üí suggest `refresh_mode => 'full'`

#### GUC-DEFAULTS ‚Äî Review dangerous GUC defaults

- `planner_aggressive = true`: auto-disables nestloop, raises work_mem for large
  deltas ‚Äî can cause unexpected spills. Evaluate whether `false` is a safer
  default for new deployments.
- `cleanup_use_truncate = true`: takes `AccessExclusiveLock`, problematic with
  PgBouncer. Evaluate changing default to `false`.
- Document trade-offs in `docs/CONFIGURATION.md` regardless of outcome.

#### BUF-LIMIT ‚Äî Change buffer hard growth limit

If refresh fails repeatedly, change buffers grow indefinitely.

- Add `pg_trickle.max_buffer_rows` GUC (default: 1,000,000)
- When a buffer exceeds the limit at refresh time, trigger FULL refresh and
  truncate the buffer; emit WARNING with buffer size and table name
- E2E test: inject repeated refresh failures; assert buffer doesn't grow beyond
  the limit

### Phase 2 ‚Äî Auto-Indexing

#### AUTO-IDX-1 ‚Äî Auto-create indexes on GROUP BY / DISTINCT columns

At `create_stream_table()` time:

- Walk the defining query's GROUP BY and DISTINCT ON expressions
- For each expression that is a simple column reference, issue
  `CREATE INDEX IF NOT EXISTS` on the stream table storage
- Skip complex expressions (function calls, casts)
- Gate behind `pg_trickle.auto_index` GUC (default `true`)
- E2E test: GROUP BY stream table; confirm index exists; verify EXPLAIN uses it

#### AUTO-IDX-2 ‚Äî Covering index on `__pgt_row_id`

For stream tables with ‚â§ 8 output columns, create:

```sql
CREATE INDEX IF NOT EXISTS __pgt_ridx_<id>
  ON pgtrickle.<st_name> (__pgt_row_id)
  INCLUDE (col1, col2, ...);
```

- Enables index-only scans during MERGE for small stream tables
- Skip for stream tables > 8 columns (covering index overhead not worth it)
- Respect `auto_index` GUC

### Phase 3 ‚Äî Test Coverage Hardening (High-Priority)

> ‚öÝÔ∏è These tests **must land before** Phases 4‚Äì9 so regressions surface immediately.

#### TG2-WIN ‚Äî Window function DVM execution tests

Currently ~5 unit tests exist but 0 DVM execution tests covering window functions.

- Add E2E tests for: `ROW_NUMBER()`, `RANK()`, `DENSE_RANK()`, `LAG()`/`LEAD()`
- Each test: INSERT cycle ‚Üí assert correct window result; then UPDATE/DELETE cycle
  ‚Üí assert window recomputed correctly
- File: `tests/e2e_window_dvm_tests.rs`

#### TG2-JOIN ‚Äî Join multi-cycle UPDATE/DELETE correctness

Current E2E join tests are INSERT-only; no UPDATE/DELETE differential cycles.

- INNER JOIN: source row updated ‚Üí stream table reflects new join value
- LEFT JOIN: right-side DELETE ‚Üí stream table row transitions to NULL
- FULL JOIN: both-side UPDATE in same cycle ‚Üí no phantom rows
- 3-table join chain: delete from middle table ‚Üí correct propagation
- File: extend `tests/e2e_join_tests.rs`

#### TG2-EQUIV ‚Äî Differential ‚â° Full equivalence validation

Add a harness that:

- Runs the same defining query in both DIFFERENTIAL and FULL mode
- Applies identical INSERT/UPDATE/DELETE sequences to source tables
- After each cycle, asserts `(DIFFERENTIAL result) EXCEPT ALL (FULL result)` is empty
- Cover: joins, aggregates, window functions, CTEs, LATERAL, TopK
- File: `tests/e2e_diff_full_equivalence_tests.rs`

### Phase 4 ‚Äî MERGE Alternatives (PH-D1)

> **Target:** sub-1% deltas against large stream tables should be cheaper than MERGE.

- Implement `DELETE WHERE __pgt_row_id IN (delta_deletes) UNION INSERT SELECT FROM delta_inserts`
- Add `pg_trickle.merge_strategy = 'auto' | 'merge' | 'delete_insert'` GUC (default `'auto'`)
- `auto` heuristic: use DELETE+INSERT when `delta_rows / target_rows < merge_strategy_threshold`
  (new GUC, default 0.01)
- Correctness: must handle pure INSERT delta, pure DELETE delta, and mixed UPDATE delta
  (UPDATE = DELETE old row ID + INSERT new row ID)
- Benchmark: `bench_ph_d1_delete_insert` in `tests/e2e_bench_tests.rs` at 1K/10K/100K
  deltas against 1M/10M targets
- SQL migration: add new GUCs to `sql/pg_trickle--0.15.0--0.16.0.sql`

### Phase 5 ‚Äî Append-Only Bypass (A-3-AO)

> **Target:** event-sourced stream tables should never pay the MERGE join cost.

- `CREATE STREAM TABLE ‚Ä¶ APPEND ONLY` declaration (new API parameter)
- Refresh path: `INSERT INTO st SELECT ‚Ä¶ FROM delta_inserts` ‚Äî no MERGE
- Only `RowExclusiveLock` required (vs `RowShareLock` held by MERGE)
- CDC heuristic: if `append_only` is not explicitly declared but pg_trickle has
  never seen a DELETE or UPDATE for this source, use the INSERT path automatically;
  fall back on first non-insert CDC event with WARNING + `pgtrickle_alert` NOTIFY
- `explain_st()` exposes `append_only_mode` (explicit/heuristic/disabled)
- Benchmark: compare INSERT throughput vs MERGE baseline for 1K/10K/100K appends
- Upgrade script: add `append_only` parameter support

### Phase 6 ‚Äî Change Buffer Compaction (C-4)

> **Target:** high-churn tables should see ‚â•50% compaction before any downstream
> processing.

- Pre-delta compaction: merge changes to the same `__pgt_row_id`:
  - INSERT + DELETE (same row ID) ‚Üí cancel out (remove both)
  - consecutive UPDATEs ‚Üí collapse to single UPDATE with latest `new_*` values
  - DELETE + INSERT (same row ID) ‚Üí net UPDATE
- Trigger compaction when `COUNT(*) FROM changes_<oid> > compact_threshold`
  (GUC `pg_trickle.compact_threshold`, default 100,000)
- Use `change_id` PK (not `ctid`) for stable row deletion under VACUUM
- Serialize compaction with the refresh advisory lock
- E2E test: high-churn source (100 updates to same 100 rows); verify compaction
  reduces buffer to ‚â§100 rows before MERGE
- Expose compaction stats in `check_cdc_health()` and `explain_st()`

### Phase 7 ‚Äî Delta Predicate Pushdown (B-2)

> **Target:** selective queries (< 10% of source rows match WHERE) should see
> 5‚Äì10√ó delta row reduction.

- Walk the OpTree for `Filter` nodes whose predicates reference only columns from
  a single source table scan
- Inject pushable predicates into the `delta_scan` CTE WHERE clause:
  - For INSERT deltas: `new_col = 'value'`
  - For DELETE deltas: `old_col = 'value'`
  - For UPDATE deltas: `new_col = 'value' OR old_col = 'value'` (both old and new
    must be checked ‚Äî a row moving *out* of the predicate window is still a deletion)
- Only push predicates that are safe to commute with joins (no correlated subs,
  no volatile functions)
- Benchmark: selective query (1% of rows match WHERE) at 100K delta ‚Äî assert filtered
  delta ‚â§ 1.5√ó expected row count
- Unit tests: pushable predicate detection + OR-split for deletions

### Phase 8 ‚Äî Algebraic Aggregate Fast-Path (B-1)

> ‚öÝÔ∏è **Hard prerequisites:** TG2-WIN, TG2-JOIN, TG2-EQUIV must all pass before
> merging B-1. The algebraic path produces subtly wrong results if the existing
> MERGE correctness tests don't already catch the edge cases.

> **Target:** high-cardinality GROUP BY (10K+ groups) at ‚â•1% change rate should
> see 5‚Äì20√ó apply-time reduction.

The B-1 work in this milestone refers to the direct `UPDATE target SET col = col + Œî`
fast-path that bypasses MERGE entirely for queries where **all** aggregates are
algebraically invertible. This is *distinct* from the auxiliary column machinery
(AVG/STDDEV/VAR via `__pgt_aux_*`) which was implemented in v0.9.0.

- **New scope in v0.16.0:** For `GROUP BY` queries where all aggregates are
  `SUM` / `COUNT` / `AVG` (algebraically invertible), replace MERGE with:
  - `UPDATE target SET sum_col = sum_col + Œîsum, count_col = count_col + Œîcount WHERE group_key = ?`
    for existing groups
  - `INSERT` for newly-appearing groups (count transitions 0 ‚Üí positive)
  - `DELETE` for groups whose count reaches zero
- `pg_trickle.aggregate_fast_path` GUC (default `true`)
- Fallback to existing MERGE path for: non-algebraic aggregates (`MIN`/`MAX`/
  `STRING_AGG`/`ARRAY_AGG`), queries with HAVING, DISTINCT aggregates
- `explain_st()` exposes `aggregate_path` field: `merge` / `direct_update`
- Property-based test: `Œîsum` is correctly accumulated across 500 random
  INSERT/UPDATE/DELETE cycles; `count = 0` groups are deleted; correct NULL handling

### Phase 9 ‚Äî Shared-Memory Template Cache (G14-SHC)

> **Depends on:** PH-D1 and B-1 settled so the MERGE template shape is stable.
> RFC from v0.15.0 G14-SHC-SPIKE is the design reference
> (`plans/performance/RFC_SHARED_TEMPLATE_CACHE.md`).

- Implement DSM + lwlock-based cache as specified in the RFC
- Each stream table gets one DSM slot keyed by `(database_oid, pgt_id, catalog_version)`
- Cache stores: the MERGE SQL string, rendering parameters, `catalog_version` at
  which it was generated
- Invalidation: bump `catalog_version` on any ALTER QUERY / DROP / reinit; each
  cache lookup checks version before returning
- PgBouncer validation: run the PB3 E2E test suite with the cache enabled;
  assert cycle correctness and cold-start latency ‚â§ 1ms after first population
- Metrics: expose `cache_hits`, `cache_misses`, `cache_invalidations` in
  `check_cdc_health()`

### Phase 10 ‚Äî Test Coverage Hardening (Medium-Priority)

#### TG2-MERGE ‚Äî MERGE template unit tests

Add unit tests for `build_merge_sql()` and `build_delta_scan_sql()` in `src/refresh.rs`:

- Correct MERGE SQL shape for INSERT / UPDATE / DELETE / mixed deltas
- `SET LOCAL enable_seqscan = off` injected for small deltas
- Partition predicate injection for partitioned stream tables
- Append-only INSERT path generates no MERGE clause

#### TG2-CANCEL ‚Äî Timeout/cancellation tests

- `SET statement_timeout = 100` before `refresh_stream_table()` on a slow query
  ‚Üí assert `ERROR` raised, no partial write to the stream table
- `pg_cancel_backend()` during active scheduled refresh
  ‚Üí assert scheduler recovers and retries next cycle; no data corruption

#### TG2-SCHEMA ‚Äî Source table schema evolution

- Column rename on source table ‚Üí stream table marks itself for reinit
- Type change (INT ‚Üí BIGINT) on source table ‚Üí handled gracefully
- Column added to source table (not in defining query) ‚Üí no-op
- Column dropped that IS in defining query ‚Üí SUSPENDED with informative error

### Phase 12 ‚Äî Performance Regression CI (BENCH-CI)

#### BENCH-CI-1 ‚Äî Benchmark harness

- Run `cargo bench` (Criterion) on a fixed hardware profile in CI
- Capture results as JSON artifacts (`target/criterion/`)
- Use `--save-baseline main` on main branch; `--baseline main` on PRs

#### BENCH-CI-2 ‚Äî Regression gate

- Parse Criterion JSON output in CI post-step
- Fail PR if any benchmark regresses > 10% vs baseline (configurable via
  `PGT_BENCH_REGRESSION_THRESHOLD` env var)
- Report regressions as PR comment with before/after throughput numbers

#### BENCH-CI-3 ‚Äî Scenario coverage

Ensure `benches/` covers at minimum:

| Scenario | Scales |
|----------|--------|
| Scan only | 1K / 10K / 100K |
| Filter (1% / 10% / 50% selectivity) | 10K / 100K |
| Aggregate (algebraic, group-rescan) | 1K / 10K groups |
| INNER JOIN (2-table, 3-table) | 10K / 100K rows |
| Window function (RANK, LAG) | 10K rows |
| CTE | 10K rows |
| TopK (ORDER BY + LIMIT) | 100K rows |
| Append-only INSERT bypass | 10K / 100K |
| Mixed workload | 100K rows |
